# 匹克球比賽平台 — 開發功能報告書

**文件版本**：1.4  
**技術棧**：Node.js、Express.js、MongoDB、Tailwind CSS  
**架構**：後台管理（內容與賽事）＋ 前台展示（賽程／即時比分／結果）＋ **裁判端**（手機更新比分）

**賽制前提**：**直接得分制**（rally scoring，每分皆由一方直接得分）；**小組賽為單循環**；**每局 15 分制並含 Deuce**；**整場可選五局三勝／三局兩勝／一局過**；賽程結構為 **小組賽 → 晉級淘汰賽**。  
**路由**：多屆大會並存時，前台以 **子路徑** 區分各屆大會（見 §6）。

---

## 1. 專案目標

建立一套**匹克球大會**專用之比賽管理與展示網站，支援：

- **安排賽程表**：小組單循環、晉級名額設定、淘汰賽 bracket 產生與手動微調。
- **計分（直接得分制）**：依賽事規則記錄局數、分數；支援進行中即時更新。
- **顯示賽程**：大螢幕／公開網址唯讀瀏覽；**比分需即時刷新，無需手動重新整理頁面**。
- **計算結果**：小組排名、晉級資格、淘汰賽晉級路徑與最終名次。
- **裁判作業**：裁判以**手機**登入，可同時負責**多場**被指派的賽事，僅能更新這些場次之比分與狀態。

## 2. 賽制與計分規則（業務定義）

以下參數於後台依「賽事／場次」可設定，並驅動計分 UI 與勝負判定。

| 項目 | 說明 |
|------|------|
| **得分方式** | **直接得分制**：每一球結束後由得分方直接加 1 分。 |
| **小組賽形式** | **單循環**（組內任兩隊各對戰一次）。 |
| **每局分數與 Deuce** | **15 分制**：目標為先取得 **15 分且領先至少 2 分** 者勝該局。當雙方比分形成 **14–14** 時進入 **Deuce**（平手延長），之後必須 **淨勝 2 分** 才結束該局（例如 16–14、17–15）。計分引擎與 UI 須支援 **Deuce 後無上限延長** 直至一方領先 2 分。 |
| **整場勝負（可選）** | **每場**可擇一：**五局三勝**、**三局兩勝**、**一局過**（單局決勝負，該局依 15 分／Deuce 規則打完即結束全場）。計分 UI 與勝負判定需依所選模式切換（例如一局過不顯示第 2、3 局流程）。 |
| **賽程階段** | **第一階段：小組賽**（單循環）；依排名與晉級名額 **第二階段：淘汰賽**。 |

## 3. 系統角色

| 角色 | 說明 |
|------|------|
| **系統管理員** | 帳號、大會／賽事設定、賽制參數、指派裁判與賽事。 |
| **賽務人員** | 建立隊伍、編排賽程、處理棄賽／改期、必要時覆寫比分（需稽核）。 |
| **裁判** | 使用手機登入；僅見**被指派的場次**（**一人可負責多場**，列表並支援快速切換）；負責**即時更新**各負責場次之比分與狀態。**不另建裁判專用權限層級**（無主裁／副裁之分、不採可配置之裁判權限矩陣）；後端僅依 **是否指派該場** 允許寫入。 |
| **公開訪客** | 透過前台 URL **唯讀**瀏覽賽程與**即時比分**（無需登入）。 |

## 4. 核心模組與功能（供 Review）

### 4.1 大會與賽事設定（後台）

- 建立／編輯**大會**（名稱、日期區間、場地列表、說明、是否啟用）。
- 建立**賽事**（例如：男雙公開組），並設定：
  - **小組賽**：組數、每組隊伍數；**單循環**自動產生對戰組合。
  - **晉級規則**：**僅「各組前 N 名」**依小組排名晉級；**不含「最佳第三名」**等跨組比序晉級。
  - **淘汰賽**：種子與對位規則、是否設季軍戰。
- **匹克球計分參數**：**15 分、須贏 2 分、14 平起 Deuce**；**整場制式**（五局三勝／三局兩勝／一局過）於 **每場比賽** 建立／編排時**個別選擇**（非依賽事預設繼承）。

### 4.2 賽程表安排（後台）

- **小組賽程**：自動產生**單循環**對戰組合，並可手動調整場地／時段／球場編號。
- **晉級後淘汰賽程**：依小組結果**自動產生對位**；允許賽務手動調整（需記錄原因者建議加稽核）。
- **賽程狀態**：未開始、進行中、已結束、延賽／取消。

### 4.3 計分 — 裁判端（手機優先）

- 裁判帳號登入後，首頁列出 **「我的場次」**（**可同時多場**，依時間／場地排序）；需支援 **場次間快速切換**（避免漏改、誤改）。
- 單一場次頁面：
  - **大字級、觸控友善**：加分鈕（隊伍 A / 隊伍 B）、必要時 undo／修正（權限與次數可限制）。
  - 依設定顯示**目前局、雙方分數、已拿下局數**；**14–14 起標示 Deuce**；達 **15 分且領先 2 分**（或 Deuce 後領先 2 分）則該局結束；整場依 **五局三勝／三局兩勝／一局過** 判定勝負（**一局過**時僅打一局即結束全場）。
- 所有寫入經 **API 驗證**：該裁判是否被指派該 `matchId`（**一對多指派**為常態）。
- **離線**（可列第二期）：現場網路不穩時之策略可另列非功能需求。

### 4.4 計分 — 後台／賽務

- 可查詢任意場次並**代為修正**（賽務／管理端；與裁判端分離）；**異動皆寫入稽核 log**。
- 小組賽全部完成後觸發或手動觸發 **晉級計算** 與 **淘汰賽對位產生**。

### 4.5 結果計算（自動為主）

- **小組賽**：勝場、敗場、局數差、得失分（直接得分制下每一分皆計入得失分較直觀）；**同分 tie-break 規則**需可設定（例如勝負關係 → 局差 → 得失分）。
- **淘汰賽**：勝方進入下一輪節點；產生冠軍／亞軍／季軍（若有）。
- 可選：**名次快照**以利爭議查核。

### 4.6 前台 — Display（公開、即時比分）

- **唯讀**、響應式（Tailwind），適合投影與手機。
- **路由與子路徑（必備）**：每一屆大會使用獨立子路徑作為前台根，例如 **`/e/{eventSlug}`** 或 **`/event/{eventId}`**（實作時擇一並全站一致），以利多屆並存、分享連結與後台預覽。該路徑下再掛載賽程、排行榜、大螢幕等子頁。**`eventSlug` 允許賽後修改**；實作上建議保留 **舊 slug 別名轉址（301）** 或同等機制，避免舊連結失效（見 §6）。
- **即時比分（必備）**：當任一場比分更新時，所有開啟前台之使用者應在**短延遲內**看到新比分，**無需手動重新整理**。
- **技術選型建議**：**WebSocket**（如 Socket.io）或 **SSE**；由後端在比分寫入後推送 `matchId` 與最新狀態；前端可依頁面訂閱 `event:{eventId}` 或僅 `match:{matchId}`。
- **大螢幕模式 — 單場專用（必備）**：提供**僅顯示一場**之全螢幕頁（隱藏其餘賽程、凸顯雙方隊名與大號比分／局數／Deuce 狀態），網址包含 **大會子路徑 + 場次識別**，例如 **`/e/{eventSlug}/screen/{matchId}`**（實際 path 於 API／路由規格訂死即可）。適合單一球場投影；仍透過即時通道訂閱該場更新。
- **賽程總覽／球場別／時間軸** 等檢視方式可並列規劃（同樣掛在大會子路徑下）。
- **排行榜／晉級圖**：與後台計算邏輯一致。
- **大會資訊區**：後台編輯之純展示內容（橫幅、公告等）。

### 4.7 後台 — Display 內容管理

- 首頁／公告／合作單位等文案與**圖片**；**圖片以「上傳檔案」為主**（後端儲存於伺服器可存取目錄或物件儲存，並記錄於資料庫），發布／排序。

### 4.8 帳號與安全（核心）

- 後台與裁判：**Session 或 JWT**（手機端 JWT ＋ HTTPS 亦常見）。
- 角色：**管理員**、**賽務**、**裁判**。裁判**不另設細部權限模組**（無主裁／副裁、無可配置 RBAC）；**裁判 API 僅校驗「已登入且該場已指派給本人」**。
- 公開頻道：**只推送公開資料**（比分、隊名、場次狀態）；管理操作仍走認證 API。
- 資安：Helmet、CORS、輸入驗證、速率限制（避免濫用比分 API）；**圖片上傳**需限制副檔名／大小／掃描策略（依部署調整）。

### 4.9 非功能需求

- **即時性**：比分推送延遲目標（例如 P95 低於 2 秒）可於實作時訂 KPI。
- 稽核：比分異動 log（含裁判／賽務身分與時間）。
- 備份：MongoDB 備份策略。

## 5. 建議資料模型（MongoDB 集合概觀）

| 集合／文件 | 主要用途 |
|------------|----------|
| `users` | 使用者、角色（admin／staff／referee）、密碼雜湊。 |
| `events` | 大會主檔（含 **slug**；**可賽後修改**，建議另表或欄位記錄 **slug 歷史／別名** 以利轉址）。 |
| `tournaments` | 賽事、階段（group／knockout）、匹克球規則參數、**晉級設定（各組前 N 名；無最佳第三名晉級）**。 |
| `groups` | 小組與所屬隊伍（若結構需要）。 |
| `teams` | 隊伍與所屬賽事／組別。 |
| `matches` | 每場：階段、對手、場地／時間、**制式**（五局三勝／三局兩勝／一局過）、**即時分數結構**（局數、目前局比分）、狀態。 |
| `match_assignments` | **裁判 ↔ 場次** 指派（**一裁判多場次**；查詢「我的場次」與 API 授權用）。 |
| `standings` | 小組排名快取或即時計算結果。 |
| `bracket_nodes` | 淘汰賽樹狀節點（可選；亦可由 matches 以 parent／round 表達）。 |
| `display_contents` | 前台展示區塊（含**上傳圖片**之檔案參照或 URL）。 |
| `audit_logs` | 比分／關鍵設定異動（建議）。 |

## 6. 技術實作要點（Express + 即時通道 + 子路徑）

- **Express**：RESTful API（`/api/admin/*`、`/api/referee/*` 需認證；公開資料可為 `/api/public/*` 或嵌在 SSR 中）；**Display 圖片上傳**可用 `multipart/form-data`（如 multer）＋ 靜態檔服務或雲端儲存。
- **前台頁面路由**：**大會子路徑為第一層**，例如 `GET /e/:eventSlug/...`，所有公開頁（賽程、排行榜、**單場大螢幕**）皆落在該前綴下，避免多屆大會混淆。**`eventSlug` 變更**：後台更新 `events.slug` 後，建議 **舊 slug 自動 301 至新 slug**（或短暫並存兩條路由指向同一 `eventId`），並避免與他屆 slug 衝突。
- **即時層**：Socket.io 與 HTTP 共用埠或反向代理；比分成功寫入 DB 後 **emit** 至 `event:{eventId}` 與（可選）`match:{matchId}`；**大螢幕單場頁**僅訂閱該 `matchId` 亦可。
- **MongoDB**：Mongoose；`matches` 需表達「第幾局、雙方分、**是否處於 Deuce 區間**、是否賽末」與局數統計，並與 15 分／領先 2 分規則一致。
- **Tailwind**：裁判頁採大按鈕、多場列表清楚可點；**大螢幕單場**用超大比分字重與高對比。
